# Implementation Phases

Rust gRPC server + CLI client. SQLite for storage via Quiver ORM. Each phase ends
with a working, testable artifact. Later phases never break earlier ones.

## Tech Stack

| Concern | Choice | Crate / Tool |
|---|---|---|

### Quiver ORM Usage Pattern

```
# Schema file: schema.quiver (at project root)

# Generate typed Rust model structs
quiver generate schema.quiver -t rust-serde -o server/src/db/
```

Generated code provides:
- Model structs with `TryFrom<&Row>` for automatic row-to-struct conversion
- `Display` and `FromStr` implementations for enums
- Serde serialization/deserialization for all structs

Connection pooling:
- `SqlitePool` with `PoolConfig` for connection management
- `pool.acquire()` returns a `PoolGuard` (auto-returns connection on drop)

Transaction pattern:
```rust
let mut conn = pool.acquire().await?;
let tx = conn.begin().await?;
// ... tx.execute(&stmt).await? / tx.query(&stmt).await?
tx.commit().await?;
```

DB URL is passed at runtime when constructing `SqlitePool::new(PoolConfig::new("file:./dev.db", 4)).await`.

## Phase 8: Integration Test Suite

**Goal:** Comprehensive end-to-end test suite that exercises every RPC through a real gRPC
server backed by a temporary SQLite database. Each test case is fully isolated -- it spins
up its own server with a fresh DB, runs the scenario, and tears down.

### Test Harness (`server/tests/harness.rs` or `tests/common/mod.rs`)

```rust
/// Spawns a tonic server on a random available port with a fresh
/// temp-file SQLite DB. Returns the client channel and a guard
/// that shuts down the server + deletes the DB on drop.
struct TestFixture {
    client: IssueTrackerClient<Channel>,
    db_path: PathBuf,       // temp file, deleted on drop
    _shutdown: oneshot::Sender<()>,
}

impl TestFixture {
    async fn new() -> Self { /* ... */ }
}

impl Drop for TestFixture {
    fn drop(&mut self) {
        // DB file cleaned up here
    }
}
```

Key properties:
- **One temp DB per test.** Created via `tempfile::NamedTempFile`. Never `:memory:` --
  we want to test real file-backed SQLite (WAL mode, locking).
- **Random port per test.** Bind to `127.0.0.1:0`, extract the assigned port.
- **Full migration applied.** The server runs its normal startup migration path
  against the fresh DB, so migration correctness is tested implicitly.
- **Graceful shutdown.** The fixture holds a `oneshot::Sender`; dropping it signals
  the server task to shut down.
- **No shared state.** Tests can run in parallel (`cargo test -j N`).

## Cross-Cutting Concerns (applied from Phase 0)

### Error Handling
- `thiserror` for domain errors (not found, validation, conflict).
- Map to gRPC status codes: `NOT_FOUND`, `INVALID_ARGUMENT`, `ALREADY_EXISTS`, `FAILED_PRECONDITION`.
- Fail-fast: no silent fallbacks.

### Database (Quiver ORM)
- ALL operations (reads included) wrapped in transactions via `pool.acquire()` -> `conn.begin()` -> `tx.commit()`. As of Phase 9, every service (component, issue, comment, hotlist, search, event log, ACL) consistently uses transactions for all DB interactions.
- `SqlitePool` with 4 connections; `PoolGuard` returns connection to pool on drop.
- Schema defined in `schema.quiver`, models generated to `server/src/db/row_mapping.rs`.
- DDL generated by quiver codegen, applied on server startup (no separate migration files).
- DB URL passed at runtime: `SqlitePool::new(PoolConfig::new("file:./data.db", 4)).await`.
- For integration tests: each test creates a temp file via `tempfile::NamedTempFile`, `init_db` creates pool and applies DDL, cleaned up on drop.

### Event Logging
- Every mutation appends to `event_log` within the same transaction.
- Structured JSON payload with before/after diff.

### Testing Strategy
- **Unit tests** (per-phase, `#[cfg(test)]` in each module): test domain logic (status machine, cycle detection, query parser) with in-memory SQLite.
- **Integration tests** (Phase 8, `server/tests/`): full gRPC round-trips through a real server across 9 test files (one per service). Each test gets its own temp-file SQLite DB (via `tempfile::NamedTempFile`), cleaned up on drop. Tests run in parallel.
- **No mocks.** Real SQLite, real gRPC, real tonic server per test.
- **No `:memory:` for integration tests.** Temp files test real file-backed SQLite behavior (WAL, locking).

### Proto Design
- Follow Google AIP conventions in proto:
  - `name` field as resource identifier
  - `FieldMask` for updates
  - `page_size` / `page_token` for pagination
  - Standard error model
